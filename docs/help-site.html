Understand · What is CP Catalog

What is CP Catalog

A self-hosted metadata catalog for BBH Capital Partners — four 360° lenses plus PII classification, organized by Project (SEI vs Non-SEI).

CP Catalog harvests metadata from your existing artifacts and systems, stores it in a dedicated Oracle schema, and presents it as a searchable, navigable map of the data estate. It is read-only — it never writes back to source systems and never runs pipelines. The catalog is a byproduct of the pipeline: ingestion runs idempotently after each dbt build.

The five inputs
  • Interface inventory (interfaces.xlsx) → Interface 360
  • SEI feed dictionary (SWP_EOD_Data_Feeds.xlsx) → Data 360
  • SEI Swagger specs (API-SPEC/) → API 360
  • PII attributes list (PII_Attributes_List.xlsx) → PII classification
  • Live: dbt manifest, Airflow metadata DB, Oracle/SQL Server schemas

The loader contract

Every connector writes only through one method — an idempotent Oracle MERGE. The protect list shields overlay/PII columns from being overwritten on a re-harvest, so business edits and PII tags survive every run.

loader._merge(table, pk, values, protect=('business_desc','is_pii',...))

SEI vs Non-SEI

project_id is a first-class dimension on every entity, resolved by one ProjectResolver so assignment is deterministic and auditable.

SourceRule
dbt modelmeta.project, else name ^sei_/^swp_ → SEI
Airflow DAGsei tag, else ^swp_/^sei_ → SEI
Oracle schemain SEI_ORACLE_SCHEMAS → SEI
API specunder API-SPEC/ → SEI
Interface systemfuzzy map: AddVantage→addvantage, Pivotal→pivotal…
To add a Non-SEI project: insert a row in the projects table (or the override CSV) and, if it needs auto-detection, add a rule in ProjectResolver. Nothing else changes.

How ingestion works

One orchestrator (run.py) runs the steps in order, each guarded (one failure won't abort the rest), ending with the PII matcher.

projects▶ oracle▶ feed_dictionary▶ dbt▶ airflow▶ interface360▶ pii_classification▶ pii_match

What each step does

  1. projects Seed/refresh the project registry (SEI + Non-SEI).
  2. oracle Harvest tables/views/columns from configured schemas; project per schema.
  3. feed_dictionary Parse the SWP feed workbook into FEED datasets + columns + enumerations (all SEI).
  4. dbt Read manifest.json: models, table lineage, compiled SQL, then column-level lineage with transform expressions.
  5. airflow Read the Airflow Postgres metadata DB: DAGs, runs, and DAG→model orchestration edges.
  6. interface360 Parse interfaces.xlsx; resolve source/target project; derive systems and routing hops.
  7. pii_classification Load the PII attribute dictionary.
  8. pii_match (last) Scan every harvested field, tag PII, write matches with the source entity's project.
Every step is idempotent — re-running is always safe. The PII matcher runs last because it must see every field every other step produced.

Run ingestion

Locally

cd ingestion
pip install -r requirements.txt
export CP_CATALOG_DB_DSN="oracle://catalog_user:pwd@localhost:1521/FREEPDB1"
export CP_CATALOG_ROOT="/path/to/CP-CATALOG"
export SEI_ORACLE_SCHEMAS="SEI_RAW,SEI_STAGE"
python -m ingestion.run

On OpenShift

oc create job --from=cronjob/cp-catalog-ingestion seed-now -n cp-catalog
oc logs -f job/seed-now -n cp-catalog

The scheduled CronJob runs daily; trigger a manual run any time. Each run refreshes the whole catalog safely.

Artifact formats

interfaces.xlsx

24 columns, row 1 = headers (exact). Key columns: Source System, Target System, Type (feed type), Feed Routing (multi-hop, --> separated), and a migration flag.

SWP_EOD_Data_Feeds.xlsx

Multi-tab: a Contents index plus one tab per feed. Each feed tab has: Position | Field Name | Field Description | Data Type/Format | Max Length | Nullable. Data types parse as NUMBER(p,s), VARCHAR2(n), DATE fmt. Coded fields like 0 (Null) 1 (Active) become enumerations. PK = position 1 + name ends _ID + not null.

swagger-spec-*.yaml

OpenAPI 3.0. The info.description carries Function point Id is 491. and the endpoint URL; error tables are Markdown (S.No | Business Exception | Error Code | Error Details). Everything under API-SPEC/ is SEI.

PII_Attributes_List.xlsx

One tab, three columns: PII Attribute | PII Components | PII Sensitivity Category. One attribute can group many components. Sensitivity drives the severity ordinal (Client Account Asset).

manifest.json

Standard dbt manifest. Per model: name, schema, compiled_code, depends_on.nodes (lineage), and config.meta.project (project hint). The compiled SQL is parsed into column-level lineage with transform expressions.
