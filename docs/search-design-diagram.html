CP Catalog — Full-Text Search: Document Design & Flow

How every artifact becomes a searchable "document", and how a result links back to the real page

1 The Document — one row per artifact in search_index

Each feed, field, loader, API, datapoint, or PII attribute is flattened into one "document". Only body_text is read by the Oracle Text index; the rest is metadata carried alongside, including the nav target that links the result back to its real module page.

Why body_text repeats the name: Oracle Text SCORE() counts term frequency. Putting the name in twice plus the description means a search for "account_number" scores the exact field higher than a feed that merely mentions account numbers in its description.

4 Query & retrieval flow — how a result links back

User types "account" in the header search → GET /search?q=account
Build Oracle Text expression with stemming: (stem(account) | account)
One query: CONTAINS(body_text, :ctx, 1) > 0 ORDER BY SCORE(1) DESC — matches across ALL artifacts at once
Each matching document returns: module, kind, name, subtitle, nav target, SCORE
UI groups by module, renders each row with a module badge + name + subtitle
User clicks → onOpen(nav) → App routes to nav.module, passes selection={tab, id}
The module opens the actual artifact using nav.id (the real key) — live page, not the index copy

5 Architecture — search_index is a derived read-model

The source tables stay the system of record. search_index is a flattened copy built at ingest, carrying enough to (a) match text and (b) navigate back. That's why it needs the sync step — and why a result links via a stored nav target, not a live foreign-key join.

Source tables — system of record
datasets · columns · feed_catalog · ldr_* · api_* · dp_registry · pii_*
↓ flatten at ingest (denormalize)
search_index + ix_search_body — derived read-model
one document per artifact · one full-text index · rebuilt/synced each load
↓ CONTAINS + SCORE (one query)
/search → ranked results
each result carries a nav target back to the live source page
Tradeoff: a derived copy can go stale, so the ingestion step re-syncs it every run. A live-join search wouldn't go stale but couldn't be full-text indexed or ranked. This is the standard search-index pattern (same reason Elasticsearch/Solr sit alongside a database).
